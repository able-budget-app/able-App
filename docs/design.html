1What Able is

Able is a budgeting app for people with inconsistent income — freelancers, creators, solo business owners, contractors. It is built around one promise: From Unable → Able.

The product is a per-deposit allocation engine with manual income/bill entry and an AI advisor (the Coach). Every time the user logs a deposit, the app splits it across buckets in a fixed order. The user enters bills, debts, income sources, and balance manually. There is no Plaid auto-sync, no spending tracking, no auto-mark-paid. The Coach is a chat that has full read access to the user's allocation state.

That's the entire product. Everything else is in service of those two loops: log deposit → see allocation and ask Coach → get specific advice.

Why it exists

The audience this app serves doesn't have a "spending problem" — they have a timing problem. Money lands on Day 1, leaks out by Day 14, and Day 30 is the panic. Generic budgeting tools (Mint, YNAB, spreadsheets) assume regular income and ask the user to track every transaction. Able inverts that: the user does almost nothing day-to-day, but every deposit is processed through a rule engine that sets aside money for taxes, bills, owner pay, debt, and a buffer before it can be spent.

Who it is for

Three working personas:

  • The Freezer — avoids opening the banking app. Wants permission and clarity.
  • The Leaker — knows roughly where money goes but can't stop it draining. Wants structure.
  • The Shame-Cycle — has tried YNAB / spreadsheets / envelopes, fell off, blames self. Wants something that survives a missed week.

Vocabulary discipline

Use these labels verbatim in any user-facing copy:

  • "Pay yourself" — not "owner pay" in UI.
  • "Extra debt payoff" — not just "debt".
  • "Savings / buffer" — never "smoothing reserve".
  • "Yours to spend freely" or "Spend freely".
Common copy mistakes
  • Wrong: "Taxes / bills / smoothing / debt / free" — wrong order, wrong vocab.
  • Wrong: "The buffer auto-releases when bills come due." It doesn't — buffer is one-way accumulation.
  • Wrong: "Plaid syncs your accounts." Not shipped. Phase B comment only.

Per-bill reservations

Money set aside for bills is tracked per-bill, not as a scalar pool. State lives at S.settings.reservations: { [billId]: amount }. The dashboard headline numbers (balance, reserved_total, available_to_spend) all derive from this map plus the user-entered balance.

The smart bill window

billsInWindow(days) is the engine's view of "what's coming up." Marking April's truck loan paid on April 25 still surfaces May 5 in the window — the next deposit reserves fresh for it. This is what makes "set it and forget it" actually work for monthly bills.

Auto month rollover

autoRolloverIfNewMonth() runs on every load path. Detects month change, clears paid flags, preserves reservations so set-aside money rolls forward, updates allocResetMonth. It does NOT auto-archive into month_history — that's a manual resetMonth() for users who want a clean record.

5Data model

Client-side state (S)

Stored as a single JSON blob in user_data.data (JSONB column). The shape is the JSON the client serializes; there is no normalized schema.

S = {
  bills:    [ { id, name, amount, dueDay, frequency, paid, ... } ],
  debts:    [ { name, balance, minPayment, ratePct, dueDay } ],
  income:   [ /* historical deposits with allocation breakdowns */ ],
  forecast: [ /* expected income */ ],
  sources:  [ /* user-defined income source labels */ ],
  obligations: [ /* off-the-top fixed allocation rules */ ],
  settings: {
    debtPct, bufPct, freePct, ownerPct,         // surplus split
    window,                                     // 7|14|21|30
    reservations: { [billId]: amount },         // per-bill earmarks
    balance,                                    // user-entered total
    allocResetMonth,                            // 'YYYY-MM' anchor
    lastOccurrenceCheck,                        // ISO date
    emailReminders: { ... },                    // 5 toggles
  },
  profile: { skipped, name, business, ... },
}

6Settings — what the user can tune

  1. Fixed allocations — off-the-top rules (taxes, contractor split, owner pay). Fire BEFORE the surplus split.
  2. Planning window — 7 / 14 / 21 / 30 days.
  3. Surplus split — sliders for pay yourself / debt / buffer / free spending. Sum ≤ 100.
  4. Income sources — user-defined dropdown labels.
  5. Email reminders — 5 toggles: bill_reminder, low_buffer, monthly_wrap, dormancy, weekly.
  6. Subscription — opens Stripe customer portal.
  7. Account — email, replay tour, contact support, close month, clear data, sign out.

7Onboarding

Triggers if the user has no bills AND has not skipped (profile.skipped !== true). 9 questions, AI-parsed by the ai-onboard Edge Function. Skip is allowed.

  1. Welcome
  2. Name + business
  3. Income payment structure
  4. Income sources list
  5. Contractor / commission splits
  6. Monthly bills (table)
  7. Debts (name, min payment, rate %, due, balance)
  8. Goals / priorities
  9. Pay yourself percentage (default 10%)

8AI Coach

A real chat with full state context. Advisory only — it does not move money.

What the Coach can see

buildCoachState() ships these fields to the model:

  • All bills, debts, settings, sources, obligations, profile
  • Headline numbers: balance, reserved_total, available_to_spend
  • reservations_by_bill — per-bill {bill, amount, reserved, days_until}
  • past_due_bills — monthly bills past due day, not paid
  • latest_deposit — most recent income with funding breakdown (funded_prior / funded_now / partial / uncovered)
  • balance_neutral flag on "Tell me where it goes" reallocations
  • Last 3 months of history + forecast data
Sync rule

Updates to buildCoachState MUST be mirrored in the system prompt — it's the only way the model knows to reference the new fields.

Proactive nudges

  1. Owner pay percentage = 0 but income has been logged.
  2. Buffer low + bills due soon.
  3. No income logged in 14+ days.

Dismissed nudges are suppressed for 12 hours (anti-nag).

Motion rules

  • Scroll reveal: .reveal fade-up, 0.6s ease, threshold 0.12.
  • Problem-beat stagger: 180ms between items. Do not loop.
  • Phone float: 6s ease-in-out. No bounce, no parallax.
  • Hover: subtle shadow + 1px lift. No color flips.
  • Allowed celebration: the checkmark pop on completion. Sparingly.

Voice + microcopy

  • Short sentences. Plain language. No em dashes.
  • Warm, calm, specific. Not hype.
  • Customer is the hero. Able is the guide.
  • Permission, not blame. "The problem was never you. It was the advice you were handed."
  • Identity language. "You are able to..." not "Able does X for you."
  • Real specifics over claims. "$500–$1,000 leaks every month" beats "save more money."
Day 1–31 anchor

Verbatim and load-bearing — never paraphrase loosely:

  • Day 1: Money comes in. Just in time. Breathe a little.
  • Day 7: A snack here, a subscription there.
  • Day 14: The leak begins.
  • Day 30: "Where did that $1,000 go?"
  • Day 31: Shame. Guilt. Avoidance.

Anti-patterns

  • Em dashes anywhere in copy. (Use periods or line breaks.)
  • "We" / "our team" voice.
  • Tool-as-hero phrasing ("Able tells you what to do").
  • Stock-photo people. Use phone mockups + abstract green gradients.
  • Multiple display faces.
  • Bouncy motion, parallax, autoplaying carousels.
  • Pricing without the "7-day free trial" line.
  • Pain framing that blames the user.

14Deployment

  • Static HTML auto-deploys from main to Netlify on every push.
  • Edge Functions deploy via scripts/deploy-functions.sh <name>. The CLI flow runs deno check locally first.
  • Database changes — paste SQL into Supabase dashboard. Files in supabase/*.sql are the source of truth.
  • Environment — Stripe keys, Anthropic key, Resend key, Supabase service role — all set in Edge Function secrets.

Health checks: functions-check.yml runs deno check on every PR; functions-healthcheck.yml pings create-checkout every 5 minutes and opens a GitHub issue on 5xx/timeout.

15Open questions / live tensions

  1. app.html is approaching the limit of "single file is simpler." At ~9.5k LOC, debugging is fine, but cognitive load grows fast. Settings and onboarding are the next extraction candidates.
  2. Plaid (Phase B). Coach state and reservations model are designed to absorb live-bank data without a rewrite. S.settings.balance can become a Plaid-derived value with minimal churn.
  3. Buffer automation. Today the buffer is one-way. The honest answer in marketing is "you transfer manually." If we ship automation, that's code → skill update → content rewrite, in that order.
  4. Pricing. Currently $14.99/mo + $129/yr (reverted from $9.99/$79). Trial-with-card is converting better than trial-no-card was at the lower price.
  5. Coach trust. The system prompt forbids the Coach from inventing actions it can't take. Any "I moved $200 to your buffer" in production is a regression.
